# 面向4C2026人工智能实践赛的个人编程学习系统方案（竞赛交付版）（Heliora）

## 摘要
本方案面向软件工程本科生，聚焦“会写代码但不会系统调试与表达”的真实痛点，设计一个以学习增益为目标的学习辅助系统。为避免技术堆砌，本项目保留三层主干：数据基础层（个人知识图谱PKG + Tree-sitter/LSP语义解析）、教学交互层（引导式纠错模块）、应用验证层（沉浸式答辩演练舱）。系统定位不是自动代写，而是辅助学生完成“定位问题-理解原理-复盘表达”的学习闭环。

在此基础上，新增“日常生活服务扩展能力”：支持本地文件自动检索、按盘符/目录配置、定时整理、手动快速上传、后续语音与实时屏幕理解扩展。通过“自研编排中枢 + MCP兼容适配 + 可替换执行器”能力组合，实现“学习主功能稳定交付，生活能力按插件渐进扩展”。

方案严格对齐4C赛事导向：强调原创、实践、可展示、可复现。文档给出可执行的里程碑、最小可行实现（MVP）、风险与备选方案，并明确所有外部数据指标与本项目自测指标的边界，确保答辩时可解释、可信、可落地。

## 1. 赛道要求与项目定位

### 1.1 赛道约束（基于公开通知）
根据高校发布的4C2026组织通知与大赛资料下载页面，人工智能应用赛道对软件类作品的底线要求包括：
1. 作品原创，禁止抄袭，违规可取消资格与奖项。
2. 软件类作品需提交源代码与源文件。
3. 参赛应按赛区提交格式与模板规范完成材料。
4. 评审关注实践能力、创新能力、协作与AI素养。

以上约束决定本项目必须做到：
1. 关键逻辑自主实现并可白盒展示。
2. 架构复杂度与本科团队交付能力匹配。
3. 以可演示原型和实测数据支撑“应用价值”。

### 1.2 项目一句话定位
为软件工程学生提供“课件-代码-报错”联动的学习辅助系统，通过启发式问答替代直接给答案，提升调试能力与答辩表达能力。

同时提供“可控、可回滚、可审计”的日常工作支持能力（文件检索、资料整理、任务提醒），形成“学习与日常双场景”的统一入口。

### 1.3 产品形态（必须可见）
1. VS Code插件：主交互入口（报错解释、引导提问、代码上下文提示）。
2. Web可视化面板：展示个人知识图谱与课件溯源链路。
3. 答辩演练页：按项目上下文生成问题并给出结构化反馈。
4. 桌面工作台（多视图）：学习视图、文件管家视图、知识图谱视图、演练视图、系统日志视图。

### 1.4 产品边界与扩展原则
1. 主功能不变：学习闭环是评审主线，不被生活扩展能力稀释。
2. 扩展可插拔：生活能力以独立模块接入，失败不影响学习主链路。
3. 高风险动作默认人工确认：删除、移动、批量重命名必须二次确认。
4. 可追溯：所有自动化操作必须具备任务ID、执行日志与回滚入口。

## 2. 核心创新（只保留一个主轴，两个支撑）

### 2.1 主轴创新
主轴创新是“基于个人知识图谱的引导式纠错交互”。

不是比拼谁生成代码更快，而是通过图谱上下文驱动提问策略，让学生先解释、再修改、再反思。

### 2.2 两个支撑创新
1. 支撑一：双引擎语义解析（Tree-sitter + LSP）
用于稳定提取代码结构与跨文件语义，解决仅靠文本切块导致的语义断裂。

2. 支撑二：沉浸式答辩演练舱
把学习过程中的代码证据与知识点映射直接转化为答辩问答训练。

### 2.3 不纳入首版评分主线的能力
为控制风险，以下能力列为扩展项，不作为首版必须完成：
1. GNN端到端训练。
2. DPO等模型微调。
3. 大规模自动标注闭环。

## 3. 系统总体设计（三层主干 + 扩展层）

### 3.1 架构概览
1. 数据基础层：
- 输入：课程讲义、示例代码、学生仓库、编译报错。
- 处理：实体抽取、AST解析、符号关系抽取、图谱写入。
- 存储：Neo4j（关系图谱）、PostgreSQL（事实/审计）、Vector Index（向量检索）。

2. 教学交互层：
- 输入：学生问题 + 当前代码上下文 + 图谱检索结果。
- 处理：引导式提问策略（澄清-假设挑战-关联-反思）。
- 输出：不直接贴完整答案，先给定位与思路，再给最小补丁建议。

3. 应用验证层：
- 输入：项目结构、历史提交、已掌握知识点。
- 输出：答辩问题、回答评分、薄弱点建议与复习路径。

4. 生活服务扩展层（新增）：
- 输入：用户配置的盘符/目录、时间策略、文件元数据。
- 处理：文件扫描、分类建议、规则整理、风险预检、任务调度。
- 输出：检索结果、整理计划、执行报告、异常告警。

### 3.2 分层模块清单（便于后续拓展）
1. Gateway层：统一API入口、鉴权、会话路由、限流。
2. Orchestrator层：任务编排与工具路由（自研编排中枢 + MCP兼容适配 + 可替换执行器）。
3. Domain层：学习域、文件域、答辩域、系统域。
4. Tool Adapter层：各类工具协议适配器与安全沙箱。
5. Data层：图数据库、向量索引、关系数据库、对象存储。
6. Observability层：结构化日志、指标、链路追踪、告警。
7. Collaboration & Dispatch层：多AI协同、任务拆分、能力匹配、队列路由、失败重试、结果聚合。

### 3.3 路由形态：多视图工作台（替代单聊天页）
1. 学习诊断视图：报错解释、引导提问、最小补丁建议。
2. 文件管家视图：扫描任务、整理计划、手动上传、执行回放。
3. 知识图谱视图：Concept-CodeUnit-BugCase关系浏览。
4. 答辩演练视图：问答训练、评分报告、薄弱点追踪。
5. 系统运维视图：任务队列、日志检索、工具调用状态。

### 3.4 图谱实体定义（首版）
1. Concept：课程知识点（来源页码、章节、关键词）。
2. CodeUnit：类/函数/文件（语言、路径、复杂度）。
3. BugCase：错误记录（错误类型、栈信息、修复状态）。
4. Rule：项目约束（命名规则、异常处理规则）。
5. Link：Concept-CodeUnit、BugCase-CodeUnit、Rule-CodeUnit。
6. FileAsset（新增）：生活文件实体（路径、类型、大小、最近访问时间、标签）。
7. TaskRun（新增）：自动化任务实例（任务类型、状态、耗时、风险等级）。

## 4. 关键实现方案（强调可做成）

### 4.1 个人知识图谱构建（MVP）
首版仅覆盖一门课程（建议《数据结构》）和一种主语言（建议Java或Python）：
1. 讲义摄入：按页切分，关键词与标题提取。
2. 代码摄入：Tree-sitter提取函数/类/调用关系。
3. 报错摄入：解析编译器日志，绑定到CodeUnit。
4. 链路生成：先规则匹配（关键词 + 结构特征），后由LLM重排。
5. 人工校验：提供简单界面允许学生修正错链路。

为什么这样设计：
1. Tree-sitter官方定位为增量解析、可容错、可在编辑中实时更新，适合学生代码经常不完整的场景。
2. LSP可提供定义跳转、引用查找等跨文件语义能力，补足Tree-sitter只做语法层的不足。

### 4.2 引导式纠错模块（MVP）
交互策略采用四步模板（每次只走一轮，避免挫败）：
1. 澄清：先让学生说明目标与现象。
2. 定位：结合图谱指出最可能问题点与证据来源。
3. 反思：提出一个必须回答的问题后再给补丁。
4. 复盘：要求学生写1条“为什么这次会错”的反思。

机制护栏：
1. 如果学生连续2次无法回答，自动降难度并提供半步提示。
2. 提供“提示强度滑块”（轻提示/标准/详细），避免过度挫败。

### 4.3 沉浸式答辩演练舱（MVP）
1. 问题生成维度：动机、架构、选型、创新、局限。
2. 每次演练10题：6题基础 + 3题追问 + 1题极端场景。
3. 评分维度：准确性、结构性、证据引用、风险意识。
4. 输出报告：高频薄弱点Top3 + 对应复习节点。

### 4.4 文件自动检索与整理模块（新增MVP）
1. 范围配置：用户可选择盘符（如D:/E:/）与白名单目录；默认排除系统目录。
2. 定时策略：支持每天/每周扫描，支持闲时执行与电量保护策略。
3. 检索能力：按文件名、后缀、时间范围、关键词、项目标签检索。
4. 整理策略：先生成“整理预案”，用户确认后再执行移动/重命名/归档。
5. 风险控制：
- 所有高风险操作（删除、覆盖、跨盘移动）默认关闭。
- 执行前生成变更清单，执行后生成可回滚日志。

### 4.5 “等不及解析时可直接上传”处理流程（新增）
1. 快速上传通道：前端支持拖拽上传文件或文件夹（限制大小与格式）。
2. 双轨处理：
- 快速轨：先做轻量索引（文件名/摘要/类型），30秒内可检索。
- 深度轨：后台异步完成语义解析与图谱链接，完成后自动刷新结果。
3. 状态可见：前端显示“待解析/解析中/可用/失败重试”四态。
4. 降级策略：解析超时则保留快速轨结果，不阻塞用户查询。

### 4.6 未来扩展接口（语音 + 实时屏幕）
1. Voice Adapter：ASR/TTS标准化接口，统一转写和播报。
2. Screen Adapter：屏幕区域采样、OCR、视觉标签抽取。
3. Event Bus：把语音事件、视觉事件统一投递到编排层。
4. 能力开关：通过配置中心按用户/设备动态启用。

### 4.7 多AI协同与任务分发（新增）
1. 协同模式（Orchestrator-Worker）：
- Supervisor Agent：负责任务拆分、状态推进、异常回退。
- Router Agent：根据意图、上下文和能力目录选择工作代理。
- Worker Agents：DebugAgent、GraphAgent、PracticeAgent、FileAgent。
- Critic Agent：对候选结果做一致性检查与证据校验后再合并输出。

2. 任务对象（统一任务协议）：
- 必填字段：task_id、parent_task_id、task_type、priority、required_capabilities、idempotency_key、payload、status。
- 生命周期：created -> routed -> queued -> running -> retrying -> completed/failed/canceled。
- 审计字段：trace_id、assigned_agent、retry_count、started_at、finished_at。

3. 分发策略（规则优先 + 能力匹配 + 动态负载）：
- 第一步规则路由：高风险文件任务必须进入 FileAgent，并强制人工确认节点。
- 第二步能力匹配：按 capability 标签筛选候选代理。
- 第三步动态调度：结合队列深度、最近成功率、超时率计算目标代理。
- 评分函数建议：$score = w_1 \cdot capability + w_2 \cdot (1-load) + w_3 \cdot success\_rate - w_4 \cdot timeout\_rate$。

4. 队列分层（任务分发）：
- realtime.queue：对话、引导提问、IDE交互（低延迟优先）。
- normal.queue：图谱检索、答辩生成、普通工具调用（吞吐优先）。
- memory.queue：记忆检索反馈、冲突复核、记忆维护（记忆专用优先级）。
- batch.queue：目录扫描、批量整理、报告生成（离峰优先）。
- deadletter.queue：超过重试上限或不可恢复失败任务。

5. 可靠性与安全：
- 重试策略：指数退避重试（建议最多3次），超过上限进入死信队列。
- 幂等控制：每个任务必须带 idempotency_key，防止重复执行。
- 可控中断：支持取消、暂停、恢复；高风险任务支持人工接管。
- 严格参数：工具调用参数采用 JSON Schema 严格校验，禁止额外字段。
- 安全模式开关：默认 `strict`；本地部署可切换 `trusted_local_max`（需显式确认与全量审计）。

6. 协议边界：
- Tool层采用 MCP（模型与工具标准连接）。
- Agent层采用 A2A 或内部 Agent Contract（跨代理协同与能力发现）。
- 对外统一通过 Orchestrator 暴露，避免客户端直连多代理。

## 5. 技术栈选型（回答“后端/前端怎么构建”）

### 5.1 后端基础栈（自研编排 + MCP兼容 + 可替换执行器）
1. 语言与运行时：Python 3.11+（主服务）、Node.js 22+（部分工具与桌面桥接）。
2. 核心库：
- FastAPI + Uvicorn：API与流式响应。
- Pydantic：配置与数据模型校验。
- SQLAlchemy + PostgreSQL（生产）/SQLite（仅开发与本地测试）：业务数据与任务状态。
- APScheduler：本地调度、周期任务与轻量任务编排。
- Celery + RabbitMQ/Redis：分布式任务队列、路由与重试（生产建议Linux环境）。
- structlog 或 loguru：结构化日志。
3. 图谱与检索：
- Neo4j：关系图谱。
- pgvector/FAISS：向量检索。
- BM25（Whoosh/Elasticsearch可选）：关键词召回。
- 混合检索策略：BM25初筛 + 向量重排 + 图谱约束。
4. 工具编排：
- Heliora Orchestrator：自研任务编排中枢。
- MCP Adapter：标准工具发现与调用。
- Execution Adapter：可插拔执行层（可兼容 OpenClaw 类方案，但不设硬绑定）。
- Internal Control Adapter：自研高权限能力（文件任务、策略引擎）。

### 5.2 前端与桌面壳
1. 核心栈：Vue 3 + TypeScript + Vite + Pinia + Vue Router。
2. UI层：基于Design Token的组件体系（统一多视图视觉与交互）。
3. 桌面分发：Electron（Windows优先），electron-builder产出安装包。
4. 通信方式：
- 渲染进程 <-> 主进程：IPC。
- 桌面壳 <-> 后端：HTTP + WebSocket。
5. 离线能力：本地缓存最近任务、索引摘要、日志快照。

### 5.3 长期记忆治理（超长上下文仅补充）
1. 分层记忆（默认不混存）：
- 语义记忆（Semantic）：稳定事实，如用户偏好与学习目标。
- 情节记忆（Episodic）：事件过程，如近期调试路径与失败原因。
- 程序性记忆（Procedural）：行为规则，如回复风格与工具调用偏好。
2. 压缩策略：
- 先抽取“事实卡片”（错误、结论、证据路径）。
- 再做结构化摘要（问题-动作-结果-反思）。
3. 检索注入：按任务类型选择上下文模板，限制注入Token预算（$K^*$ 约束）。
4. 证据优先：模型回答优先引用可追溯节点，无证据则明确不确定。
5. 成本护栏：超过阈值时自动降级到“摘要 + 关键证据”模式。

### 5.4 多AI协同实施顺序（落地建议）
1. 第一步：单 Supervisor + 2 个 Worker（DebugAgent、GraphAgent），先打通任务拆分与结果聚合。
2. 第二步：接入任务队列和优先级路由（realtime/normal/memory/batch），加入重试与死信队列。
3. 第三步：新增 Critic Agent 与人工接管节点，保证高风险动作可控。
4. 第四步：接入能力发现（MCP资源目录 + Agent Capability Registry），支持动态扩展代理。
5. 第五步：补齐监控面板（队列长度、成功率、超时率、平均耗时、代理负载热图）。

### 5.5 配置路径与跨电脑迁移
1. 配置分层：
- app-config.yaml：全局配置（端点、开关、路径策略）。
- user-config.yaml：用户偏好（盘符、扫描计划、提示强度）。
- secrets.env：密钥与令牌（不入库）。
2. 路径策略：统一使用逻辑别名（WORKSPACE_ROOT、DATA_ROOT），避免硬编码绝对路径。
3. 迁移步骤：
- 导出：配置文件 + 本地数据库快照 + 图谱导出文件。
- 导入：环境检测 -> 路径重映射 -> 依赖自检 -> 数据校验。
4. 兼容机制：保留schema版本号，启动时自动迁移并生成迁移报告。

### 5.6 关键异常日志格式（线上排障）
建议采用JSON Lines，一行一事件，核心字段如下：
1. timestamp：ISO8601时间。
2. level：DEBUG/INFO/WARN/ERROR/FATAL。
3. trace_id / span_id：链路追踪ID。
4. module：模块名（gateway/orchestrator/file_service等）。
5. task_id：任务ID。
6. user_id_hash：脱敏用户标识。
7. action：动作名（scan/start/move_file等）。
8. status：success/fail/retry。
9. error_code：标准错误码。
10. error_message：错误摘要（脱敏）。
11. duration_ms：耗时。
12. extra：扩展字段（盘符、文件数、策略版本）。

日志规范：
1. 禁止明文记录隐私路径和敏感内容。
2. 统一错误码字典，便于告警聚合。
3. 前端与后端共享trace_id，支持端到端排障。

## 6. 可行性与里程碑

### 6.1 开发阶段（建议4个月）
1. 第1月：数据基础层
- 完成讲义摄入、Tree-sitter解析、Neo4j建模。
- 交付物：可查询的最小图谱Demo。

2. 第2月：交互层
- 完成插件聊天面板、四步引导策略、日志记录。
- 交付物：可在IDE内完成一次引导式修错闭环。

3. 第3月：答辩演练层
- 完成题目生成与评分报告。
- 交付物：可展示一次完整答辩模拟。

4. 第4月：整合与打磨
- 压测、体验优化、文档与视频制作。
- 交付物：参赛版安装包、演示视频、说明书。

5. 并行扩展（每月穿插）：文件管家MVP
- 完成盘符配置、定时扫描、整理预案与手动上传双轨处理。
- 交付物：文件视图可完成一次“扫描-建议-确认-执行-回滚”闭环。

### 6.2 单人执行优先顺序
1. 先打通主链路：插件报错 -> 后端处理 -> 引导回复。
2. 再补图谱链路：课程资料 -> 代码解析 -> 错误关联。
3. 然后补演练与文件能力：答辩演练最小版 + 文件安全闭环。
4. 最后做稳定性：异常处理、日志、演示脚本。

### 7.1 原创性

## 9. 风险与备选方案

### 9.1 技术风险
1. 图谱链路误连：
- 对策：引入人工确认界面；低置信度链路不进入主流程。
2. 检索延迟偏高：
- 对策：课件和代码分层索引；热点查询缓存。
3. LLM幻觉：
- 对策：回答必须附来源节点；无证据时明确提示“不确定”。
4. 文件整理误操作：
- 对策：默认预演模式、二次确认、可回滚日志。

### 9.2 性能风险
1. 本地设备算力不足：
- 对策：本地做解析，云端做生成；提供离线简化模式。
2. 大目录扫描耗时过长：
- 对策：增量索引、分片扫描、时间窗口调度。

### 9.3 用户接受风险
1. 学生反感被连续提问：
- 对策：提供提示强度滑块与一键求助模式。
2. 用户担心自动整理不安全：
- 对策：默认只建议不执行，提供“仅预览模式”。

5. 第5分钟：展示答辩演练结果、量化提升与风险控制设计。

## 12. 结论
本方案将原先“多技术并列”的写法重构为“学习主线 + 生活扩展 + 可验证交付”的竞赛形态，核心优势在于：
1. 主线清晰：以学习增益为目标，而非代码代写。
2. 工程可做：采用模块化分层与MVP分阶段推进，避免PPT造车。
3. 扩展性强：可插拔执行层可持续接入语音与屏幕能力。
4. 答辩友好：每个结论都能回到可展示的证据与指标。

该版本适合作为4C2026人工智能实践赛的正式设计说明书底稿，可直接扩展为校赛/省赛提交文档。
