# Heliora 当前阶段手把手执行指南

你现在不用再想架构细节，先把下面流程跑通。

快速跳转:
1. 如果你是“先连上再干活”，先看 [SSH快速连接与操作速查](./SSH快速连接与操作速查.md)。
2. 如果你要看完整阶段记录和配置格式说明，先看 [配置文件选择与同步总记录_260401](./配置文件选择与同步总记录_260401.md)。

这份指南的目标只有一个：
1. 把后端服务稳定跑起来。
2. 能访问 health 接口。
3. 能跑通基础冒烟测试。

---

## 0. 先理解原理（白话）

你现在的开发链路分 3 层：

1. 基础设施层（Docker 容器）
   - PostgreSQL、Redis、Neo4j、RabbitMQ、MinIO。
   - 这是“地基”。地基没起，后端会跑得不稳。

2. 应用层（FastAPI）
   - 代码在 `/home/heliora/heliora_backend`。
   - 用虚拟环境 `.venv` 运行，不用系统 Python。

3. 验证层（curl/脚本/浏览器）
   - 先在虚拟机里用 `curl` 验证。
   - 再用 Windows 浏览器走端口转发验证。

你之前报错的本质：
1. 服务没启动成功，所以 `/health` 连不上。
2. 把 HTTP 路径当 shell 命令执行了。
3. 虚拟环境激活前后命令混用了。

---

## 1. 非常重要的一个点

你 Windows 里的 conda 环境和虚拟机里的 `.venv` 是两套系统。

结论：
1. Windows 里 `conda activate Heliora` 不会让虚拟机自动生效。
2. 虚拟机内必须执行：`source .venv/bin/activate`。

## 2. 本阶段固定操作（一步一步抄）

请开两个 SSH 终端。

终端 A 用来启动服务（保持不关）。
终端 B 用来做测试验证。

### 步骤 0（重启后推荐）：30 秒恢复后端

```bash
cd /home/heliora/heliora_backend
chmod +x scripts/start_api_bg.sh scripts/stop_api.sh scripts/start_worker_bg.sh scripts/stop_worker.sh
bash scripts/start_api_bg.sh
curl http://127.0.0.1:8000/health
```

成功信号：
1. `start_api_bg.sh` 显示 `API started in background.`。
2. 看到 `Worker started in background.` 或 `Worker already running`。
3. `curl` 返回健康 JSON。

说明：
1. 你这次报错 `curl: (7) Failed to connect`，就是重启后 API 进程未运行。
2. 以后重启后先执行这 4 行，通常即可恢复。

### 步骤 0.1（Day-3）确认 worker 在持续消费

```bash
cd /home/heliora/heliora_backend
bash scripts/check_services.sh

```

成功信号：
1. `check_services.sh` 输出 `worker running`。
2. `.worker.log` 持续出现 `no queued task` 或 `consumed task=...`。

## 3. 如果失败，按这个表看

### 情况 A：`python: command not found`

说明：你没激活虚拟环境。

处理：
```bash
cd /home/heliora/heliora_backend
python3 -m venv .venv
source .venv/bin/activate
python -V
```

### 情况 B：`pytest: command not found`

说明：依赖没装进 `.venv`。

处理：
```bash
cd /home/heliora/heliora_backend
source .venv/bin/activate
pip install -r requirements.txt
pytest
```

### 情况 C：浏览器 `ERR_CONNECTION_RESET` 或连不上

先判断是不是后端没启动：

```bash
# 在终端 B
curl http://127.0.0.1:8000/health
```

1. 如果这里失败：说明终端 A 的 `python main.py` 没跑起来。
2. 如果这里成功但浏览器失败：检查 VirtualBox 端口转发 `8081 -> 8000`。

### 情况 D：容器服务异常

```bash
cd /home/heliora/heliora_backend
docker compose ps
docker compose logs --tail=100
```

### 情况 E：`start_api_bg.sh` 后出现 `.env: line 1: $'Backend\\r': command not found`

说明：
1. 这是旧版 worker 启动脚本用 Bash `source .env` 导致的解析问题（CRLF + 空格值触发）。

处理：
```bash
# 先在 Windows 同步最新后端目录到虚拟机
rsync -avz --delete --exclude ".venv" --exclude "__pycache__" --exclude ".pytest_cache" -e "ssh -p 2026" /e/Zero9/Heliora/heliora_backend/ heliora@127.0.0.1:/home/heliora/heliora_backend/

# 再在虚拟机重启 API + worker
cd /home/heliora/heliora_backend
bash scripts/stop_worker.sh
bash scripts/start_api_bg.sh
bash scripts/check_services.sh
```
