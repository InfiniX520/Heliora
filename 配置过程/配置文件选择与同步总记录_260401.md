## 3. 为什么别人用 json，我们用 .env

这是两种用途不同的配置方式，不是谁绝对更高级。

### .env 适合什么

1. 环境变量类配置：地址、端口、账号、开关、阈值。
2. 部署时按环境覆盖：dev/staging/prod 可无代码切换。
3. 与 Docker、CI/CD、云平台天然兼容。
4. 对密钥类字段更友好（可通过系统环境注入，避免入库）。

### json 适合什么

1. 结构化配置：层级、列表、映射关系多时更清晰。
2. 静态业务数据：规则库、模板、路由表、角色描述等。
3. 需要批量编辑和可视化时，json 更直观。

### 二者差异（核心）

1. 目标不同：`.env` 偏“运行环境参数”，`json` 偏“结构化数据”。
2. 可覆盖性：`.env` 更适合按机器/环境覆盖，json 通常是仓库内固定文件。
3. 安全性：`.env` 更容易与密钥管理体系结合；json 容易被误提交敏感信息。
4. 可读性：复杂结构 json 更好，简单键值 `.env` 更轻。

## 4. 这个项目里哪个更好

结论：采用“`.env` 为主，json 为辅”的混合方案最稳。

建议落地：
1. `.env`：存运行参数与环境差异项（当前已采用）。
2. `json`：存结构化业务配置（例如复杂策略映射、静态模板）。
3. 不在 json 中存密码、密钥、令牌；敏感配置统一走 `.env` 或系统环境变量。

当前项目继续保留 `.env` 是正确的，不需要为了“看起来高级”而全部改成 json。

## 5. 当前推荐配置基线

建议保持如下关键键位：
1. `TASK_QUEUE_BACKEND=memory`（本地默认稳态）
2. `TASK_RETRY_MAX_ATTEMPTS=2`
3. `TASK_RETRY_BASE_DELAY_SECONDS=1.0`
4. `TASK_RETRY_MAX_DELAY_SECONDS=30.0`
5. `TASK_RETRY_BACKOFF_FACTOR=2.0`
6. `TASK_QUEUE_FAIL_OPEN=true`
7. `TASK_EVENTS_PERSISTENCE_ENABLED=true`
8. `TASK_EVENTS_SQLITE_PATH=.data/task_events.db`
9. `RABBITMQ_URL=amqp://heliora:***@127.0.0.1:5672/`
10. `IDEMPOTENCY_TTL_SECONDS=86400`
11. `CHAT_MAX_CONTENT_CHARS=4000`
12. `MEMORY_MAX_QUERY_CHARS=512`

需要 RabbitMQ 联调时，仅切换：
1. `TASK_QUEUE_BACKEND=rabbitmq`
