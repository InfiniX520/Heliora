# Heliora 多智能体协同与任务分发设计

> 文档目标: 明确“多智能体协同 + 任务分发”的可实现架构、调度口径与迁移计划。
> 关联文档: [01-系统架构设计](01-系统架构设计.md), [04-长期记忆系统设计](04-长期记忆系统设计.md), [06-统一口径总索引](../04-技术规范/06-统一口径总索引.md)

---

## 1. 设计结论（本阶段）

1. 采用“成熟开源底座 + 自研策略层”的路线。
2. 编排能力使用工作流框架模式（路由、并行、Orchestrator-Worker、Critic 复核），不绑定单一厂商实现。
3. 任务分发使用消息队列实现优先级隔离与重试治理（realtime / normal / memory / batch）。
4. 项目代码保持独立实现；外部项目仅做思路参考，不直接复制实现代码。

---

## 2. 范围与非范围

### 2.1 范围

1. 对话请求拆分为父任务与子任务。
2. 多角色智能体协同执行（Supervisor / Router / Worker / Critic）。
3. 分队列调度、超时控制、重试退避、幂等去重。
4. 任务全链路可观测（trace_id、状态流转、失败原因）。

### 2.2 非范围

1. 不在本阶段实现复杂跨地域调度。
2. 不在本阶段引入多租户资源配额系统。
3. 不在本阶段实现全自动策略学习（仅保留指标与人工调参入口）。

---

## 3. 组件与职责

| 组件 | 职责 | 输入 | 输出 |
|---|---|---|---|
| Orchestrator | 会话入口、父任务创建、结果聚合 | 用户请求、上下文 | 父任务、用户响应 |
| MultiAgentHub | 子任务拆分、角色协同、复核编排 | 父任务 | 子任务计划、复核结果 |
| TaskDispatcher | 路由、入队、重试与回退控制 | 子任务 | 队列任务、状态事件 |
| WorkerAgent | 执行单一职责任务（检索/生成/工具调用） | 子任务 | 候选结果 + 证据 |
| CriticAgent | 一致性、证据完整性与安全策略检查 | 候选结果 | 通过/打回/降级建议 |

协同原则:
1. Worker 不直接面向用户返回最终答案。
2. Critic 未通过时，任务进入 retrying 或 failed，不可直接 completed。
3. 所有结果必须附带 evidence 与 trace_id。

## 4. 任务模型与状态机

### 4.1 任务信封（Task Envelope）

```json
{
  "task_id": "tsk_20260329_0001",
  "parent_task_id": "ptsk_20260329_0001",
  "trace_id": "trc_8f2b0f",
  "task_type": "code_debug",
  "priority": "P1",
  "deadline_ms": 3000,
  "idempotency_key": "sessA-msg102-code_debug",
  "payload": {},
  "context": {
    "session_id": "sessA",
    "user_id": "u123",
    "scope": "project"
  },
  "routing_hints": {
    "agent_roles": ["router", "worker_backend", "critic"],
    "tool_tags": ["lsp", "repo_read"]
  },
  "retry": {
    "max_attempts": 2,
    "backoff_ms": [500, 1500]
  },
  "security": {
    "allowed_tools": ["read", "search", "analyze"],
    "write_requires_confirm": true
  }
}
```

### 4.2 统一状态机

与统一口径文档保持一致：

1. `created -> routed -> queued -> running -> completed`
2. `created -> routed -> queued -> running -> retrying -> running`
3. `running -> failed`
4. `queued/running -> canceled`

约束:
1. 状态变更必须记录到任务事件流（含时间戳、操作者、错误码）。
2. `completed` 与 `failed` 为终态，不允许回写到 `running`。

## 5. 路由与队列策略

### 5.1 队列定义

| 队列 | 典型任务 | SLA | 说明 |
|---|---|---|---|
| realtime.queue | 对话首轮定位、关键纠错提示 | <= 3000ms | 高优先级、短超时 |
| normal.queue | 常规检索与解释任务 | <= 15000ms | 默认队列 |
| memory.queue | 记忆检索反馈、冲突复核、记忆维护任务 | <= 5000ms | 记忆专用队列 |
| batch.queue | 批量分析、总结、离线整理 | <= 300000ms | 可延迟执行 |

### 5.2 路由规则

1. `priority=P0/P1` 默认进入 realtime.queue。
2. 含大量上下文检索但非阻塞交互任务进入 normal.queue。
3. 记忆相关异步任务（feedback/consolidate/conflict_review）进入 memory.queue。
4. 报告生成、批处理导入、离线压缩任务进入 batch.queue。
5. 路由结果由 Router 生成，Dispatcher 最终裁决并记录原因。

### 5.3 重试与幂等

1. 网络抖动、上游 5xx、瞬态限流可重试。
2. 参数非法、策略拒绝、证据缺失不重试。
3. 依赖 `idempotency_key` 防止重复执行副作用。
4. 超过最大重试次数进入 failed，并返回可执行降级建议。

### 5.4 安全策略模式开关

统一安全模式:
1. `strict`（默认）：执行严格策略，越权工具与未确认高风险写操作会被 Critic 打回。
2. `trusted_local_max`（可选）：面向本地部署的高权限模式，允许更高工具权限与策略覆盖，但必须满足“本地运行 + 显式确认 + 全量审计”。

模式约束:
1. `trusted_local_max` 仅允许在本机环境启用，不作为远程默认模式。
2. 启用时必须写入审计事件（操作者、时间、原因、任务ID、trace_id）。
3. 即使在 `trusted_local_max` 下，危险批量操作仍要求最少一次显式确认。

## 6. 协同执行模板

### 6.1 标准模板

1. Supervisor 接收父任务并拆分子任务。
2. Router 根据任务类型、SLA、工具需求分配 Worker。
3. Worker 并行执行并返回候选结果与 evidence。
4. Critic 执行一致性、证据完整性、安全策略复核。
5. 通过则聚合输出；不通过则打回重试或降级。

### 6.2 复核判定

通过条件:
1. 证据链完整（来源、时间、关联上下文可追踪）。
2. 与策略约束不冲突（`strict` 下按严格权限；`trusted_local_max` 下按本地高权限白名单）。
3. 输出可执行（包含下一步动作，不仅是结论）。

打回条件:
1. 证据不足或证据互相矛盾。
2. 触发策略红线（`strict` 下越权工具；任意模式下未确认高风险写操作）。
3. 与用户目标偏离。

---

## 7. 可观测性与审计

1. 统一追踪: 全链路强制 `trace_id`。
2. 事件日志: `task_events` 记录状态流转、重试、降级、取消。
3. 指标看板:
   - 完成率、失败率、重试率
   - 队列等待时长 P50/P95
   - 首轮响应延迟
   - Critic 打回率
4. 审计要求: 用户可查询任务编号与关键决策节点。

## 8. 实施计划（P0-P2）

### P0（接口冻结）

1. 固化任务信封结构与状态机。
2. 先串行执行，保留并行字段与路由字段。
3. 建立最小任务事件表与监控埋点。

### P1（生产可用）

1. 接入三队列调度与重试退避。
2. 引入 Critic 复核闸门。
3. 实现优先级抢占与超时降级。

### P2（能力增强）

1. 引入动态路由策略（基于历史质量与成本）。
2. 支持更细粒度的工具权限沙箱。
3. 支持复杂父子任务图与分支聚合优化。

---

## 9. 验收标准

1. 实时任务在 `realtime.queue` 的 P95 满足 <= 3000ms。
2. 任一任务都可追踪完整状态流转与失败原因。
3. Critic 打回后不出现“未复核直接完成”路径。
4. 重试场景无重复副作用（幂等验证通过）。
5. 取消任务后不再产生新工具调用。
6. `strict` 与 `trusted_local_max` 两种模式切换后，策略行为与审计记录符合预期。

---

## 10. 与当前阶段目标对齐

1. 保持“可运行最小系统”目标，不引入过度复杂的调度框架耦合。
2. 与长期记忆、对话策略、文件安全流程保持同一治理口径。
3. 为后续 Persona 能力、工具生态扩展预留稳定编排接口。
