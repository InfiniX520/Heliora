# Heliora 工程与代码规范总则

> 适用范围: Heliora 独立仓库（前端、后端、插件、脚本）  
> 关联文档: [[01-前端开发规范]], [[02-后端开发规范]], [[04-配置管理规范]], [[Heliora项目方案]]

---

## 1. 目的与定位

本总则用于统一 Heliora 独立仓库的工程实践，解决以下问题：
1. 方案与实现脱节。
2. 多来源文档混用导致口径不一致。
3. 未先立规范、后续开发质量不可控。

本总则优先级高于子规范。若子规范冲突，以本总则为准。

---

## 2. 独立化与合规借鉴原则

### 2.1 资产边界

1. Heliora 为独立产品与独立仓库，核心代码、提示词、流程编排、UI 资产均需自研。
2. 外部项目仅可作为架构与流程参考，不作为直接复制来源。
3. 任何第三方代码、素材、模型配置接入前必须完成许可证审查。

### 2.2 可借鉴与禁止项

可借鉴：
1. 分层思想（Gateway / Orchestrator / Domain / Adapter / Data）。
2. 模块边界定义、错误处理模式、日志字段设计思路。
3. 工具调用协议形态（如 MCP 兼容方式）与任务编排思想。

禁止项：
1. 直接拷贝第三方仓库业务代码、UI 资源、角色素材、文案与专有配置。
2. 复制第三方私有协议相关实现并作为本项目核心能力。
3. 将未审查许可证的代码进入主分支。

### 2.3 外部依赖引入流程

1. 提交依赖引入申请（用途、替代方案、许可证类型、是否静态链接）。
2. 完成技术评审 + 合规评审。
3. 在仓库记录三项信息：
   - 依赖名称与版本。
   - 许可证与来源链接。
   - 引入理由与责任人。

## 3. 范围冻结与里程碑策略

### 3.1 P0（必须完成，竞赛主线）

1. 引导式纠错闭环（澄清、定位、反思、复盘）。
2. 课程-代码-报错三联图谱最小可用链路。
3. 答辩演练基础版（题目生成 + 结构化反馈）。
4. 文件管家安全模式（扫描、预案、确认、执行、回滚）。
5. 长期记忆最小闭环（写入门控、检索注入、evidence绑定、审计与回滚）。

### 3.2 P1（增强）

1. 提示强度滑块与自动降难度策略。
2. 图谱可视化增强与检索优化。
3. 任务调度稳定性与性能优化。
4. 长期记忆混合检索（结构化+向量+图谱）与五区间冲突判别。

### 3.3 P2（扩展）

1. 个性化学习支持增强。
2. 语音与实时屏幕理解。
3. 高级个性化体验能力。

不纳入当前交付主线：社区化、娱乐化模块、与学习闭环弱相关的扩展功能。

## 4. 仓库结构规范（独立 Heliora）

推荐结构：

```text
heliora/
  apps/
    desktop/            # Electron + Vue 主应用
    vscode-extension/   # VS Code 插件
  services/
    api/                # FastAPI 主服务
    worker/             # 异步任务
  packages/
    core/               # 领域模型、配置、日志、错误码
    adapters/           # MCP/LSP/文件系统等适配器
    sdk/                # 前后端共享协议与类型
  docs/
    architecture/
    standards/
    delivery/
  scripts/
  tests/
```

说明：
1. 新功能先归类到已有模块，不允许随意新增平级目录。
2. 目录命名统一小写短横线，Python 包名统一小写下划线。
3. 跨端共享结构放在 packages，避免复制粘贴。

## 5. 分支与提交规范

### 5.1 分支模型

1. main: 可发布分支。
2. develop: 日常集成分支。
3. feature/*: 功能分支。
4. fix/*: 缺陷修复。
5. release/*: 发布准备。

### 5.2 提交信息

采用 Conventional Commits：

```text
feat: 新增引导式纠错状态机
fix: 修复文件预案回滚路径错误
docs: 更新独立化合规约束
refactor: 拆分图谱检索服务
test: 补充文件扫描边界测试
```

要求：
1. 一次提交只做一件事。
2. 提交信息必须能反映用户价值或风险修复。
3. 禁止“update”、“改一下”等无效描述。

---

## 6. 代码风格与质量门禁

### 6.1 前端

1. TypeScript 开启严格模式。
2. ESLint + Prettier 必须通过。
3. 组件命名 PascalCase，组合式函数 useXxx。
4. 样式优先 Design Token 与变量，不写魔法值。

### 6.2 后端

1. Python 3.11+，统一类型标注。
2. Ruff/Black（或同等工具）通过。
3. API 层、服务层、仓储层职责清晰，不跨层访问。
4. 异常统一转换为标准错误码，不直接抛裸异常给前端。

### 6.3 通用门禁

合并前必须满足：
1. Lint 通过。
2. Type Check 通过。
3. 单元测试通过。
4. 核心模块覆盖率不低于 70%。
5. 文档同步更新（若影响架构、接口、配置）。
6. 若涉及记忆链路，需附删除SLA与回滚演练记录。

---

## 7. API 与数据契约规范

1. 接口先定义契约（请求、响应、错误码）再开发。
2. 所有接口返回统一结构：code、message、data、trace_id。
3. 破坏性变更必须升级版本并提供迁移说明。
4. 关键链路必须携带 trace_id，支持端到端排障。

---

## 8. 安全与隐私规范

1. secrets 不入库，统一走环境变量或密钥管理。
2. 日志默认脱敏，不记录明文隐私内容。
3. 文件操作默认预演模式，高风险操作必须二次确认。
4. 引入第三方服务前，明确数据出境与合规边界。
5. 记忆删除默认 soft delete，并保证索引清除闭环SLA <= 5分钟。

---

## 9. 文档驱动开发机制

### 9.1 变更触发

以下变更必须同步更新文档：
1. 架构分层调整。
2. 新增外部依赖。
3. API 契约变化。
4. 配置项新增或默认值变化。
5. 记忆阈值策略、冲突状态机或删除SLA变化。

### 9.2 最小文档包

每个里程碑至少维护：
1. 目标与范围说明。
2. 接口清单。
3. 风险清单。
4. 验收标准与结果记录。

---

## 10. 决策原则（遇到冲突时）

1. 主线优先于扩展。
2. 合规优先于速度。
3. 可演示优先于技术炫技。
4. 可回滚优先于自动化激进策略。

本总则自发布起生效，作为 Heliora 独立仓库的工程基线。
