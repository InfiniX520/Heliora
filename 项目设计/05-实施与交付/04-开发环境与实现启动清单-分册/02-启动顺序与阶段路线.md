# 启动顺序与阶段路线

> 来源: [04-开发环境与实现启动清单](../04-开发环境与实现启动清单.md)
> 目的: 聚焦开发阶段推进路径、验收信号和风险处理。

---

## 4. 启动顺序（Day-0 到 Day-2）

### Day-0: 环境可运行

1. 创建并激活 Python 虚拟环境。
2. 安装后端依赖并可启动基础 API。
3. 拉起 PostgreSQL、Redis、Neo4j、RabbitMQ、MinIO。
4. 完成健康检查接口与日志 trace_id 输出。

验收信号:
1. API 健康检查返回成功。
2. 依赖服务连接全部通过。
3. 日志可看到统一 trace_id。

### Day-1: 接口骨架

1. 按 [07-后端接口契约草案](../../04-技术规范/07-后端接口契约草案.md) 建立空实现路由。
2. 返回统一响应结构与标准错误码。
3. 补齐 Idempotency-Key 校验中间件（有副作用接口）。

验收信号:
1. `/api/v1/chat`、`/api/v1/memory/retrieve`、`/api/v1/tasks/submit` 可调用。
2. 非法参数能返回约定错误码。

### Day-2: 任务流最小闭环

1. 跑通 `created -> routed -> queued -> running -> completed/failed`。
2. 先接 `realtime.queue` 与 `normal.queue`，并预留 `memory.queue`、`batch.queue`。
3. 打通最小审计事件落库（task_events）。

验收信号:
1. 任务可查询状态。
2. 失败任务可看到原因和重试计数。

## 5. 实现阶段（长期最优路径）

### 阶段A：接口与模型冻结
1. 冻结 MemoryRecord、TaskEnvelope、PersonaProfile 数据模型。
2. 冻结 `/api/v1` 路径口径和错误码。

### 阶段B：长期记忆主链路
1. 写入门控（`tau_write(d)`）。
2. 检索重排（结构化 + 向量 + 图谱扩展）。
3. 冲突判别（五区间 Lambda）。

### 阶段C：多智能体协同
1. Router + Worker + Critic 最小组合。
2. 四层队列（realtime/normal/memory/batch）。
3. 幂等、重试、取消、死信。

### 阶段D：Persona 与角色卡
1. PersonaProfile v1 落库。
2. 角色卡 V1/V2 导入导出适配。
3. 未知字段透传与可逆导入导出。

### 阶段E：安全模式与治理
1. strict 与 trusted_local_max 双模式切换。
2. 审计与回滚闭环验证。
3. 删除SLA与冲突未决时长监控。

## 6. 每阶段固定检查项

1. 术语与阈值是否仍与 [06-统一口径总索引](../../04-技术规范/06-统一口径总索引.md) 一致。
2. 接口是否仍与 [07-后端接口契约草案](../../04-技术规范/07-后端接口契约草案.md) 一致。
3. 记忆逻辑是否仍与 [大模型记忆实现方案.md](../../../大模型记忆实现方案.md) 一致。
4. 安全策略是否遵守 strict 默认、trusted_local_max 显式确认。

---

## 7. 常见卡点与处理

1. 卡点: 本地依赖版本冲突。
   - 处理: 统一通过 Docker 固定版本，避免系统全局安装混用。
2. 卡点: 接口可用但业务逻辑未闭环。
   - 处理: 先保证状态机闭环，再补算法细节。
3. 卡点: 高权限模式误用到常规开发。
   - 处理: 默认 strict，trusted_local_max 仅在专项调试窗口开启。
