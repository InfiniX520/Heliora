# 虚拟机实操清单（可直接执行）

> 来源: [04-开发环境与实现启动清单](../04-开发环境与实现启动清单.md)
> 目的: 将 VM 实操步骤独立，避免被开发进度和策略说明干扰。

---

## 9.0 前提

1. Ubuntu 虚拟机已启动。
2. 已可通过 Git Bash SSH 登录。
3. 精简版 Win11 无 OpenSSH 客户端时，使用 Git Bash SSH 与 JetBrains 内置 SSH 即可，不影响开发。

---

## 9.1 先建 PyCharm 项目骨架（建议先做）

结论:
1. 建议先在 PyCharm 建项目骨架，再启动容器。
2. 容器可以先起，但如果没有项目骨架与 `.env`，后续接口联调会反复改动。

最小骨架建议:
1. `backend/app/main.py`
2. `backend/app/api/`
3. `backend/app/core/`
4. `backend/tests/`
5. `backend/requirements.txt`（或 `pyproject.toml`）
6. `.env.example`
7. `docker-compose.yml`

路径一致性检查:
1. 如果在 PyCharm 创建了 `docker-compose.yml`，必须确认它在虚拟机当前目录可见。
2. 在 SSH 终端执行 `pwd && ls`，确认与 PyCharm 项目目录一致。
3. 若不一致，优先在虚拟机内维护代码目录，PyCharm 用 SSH Interpreter 连接该目录。

## 9.2 首次登录后基线加固

在 Git Bash 执行:

```bash
ssh heliora@127.0.0.1 -p 2026
```

在 Ubuntu 虚拟机内执行:

```bash
passwd
sudo apt update && sudo apt -y upgrade
sudo apt install -y git curl wget ca-certificates gnupg lsb-release jq make build-essential unzip zip tree htop net-tools
```

成功信号:
1. `passwd` 修改成功。
2. `apt upgrade` 无报错结束。

## 9.3 安装 Docker 与 Compose

在 Ubuntu 虚拟机内执行:

```bash
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER
newgrp docker
docker --version
docker compose version
```

成功信号:
1. `docker --version` 输出版本号。
2. `docker compose version` 输出版本号。

## 9.4 拉起基础依赖服务（PostgreSQL/Redis/Neo4j/RabbitMQ/MinIO）

在 Ubuntu 虚拟机内执行:

```bash
mkdir -p ~/heliora-infra && cd ~/heliora-infra
cat > docker-compose.yml <<'YAML'
services:
   postgres:
      image: postgres:15
      container_name: heliora-postgres
      restart: unless-stopped
      environment:
         POSTGRES_DB: heliora
         POSTGRES_USER: heliora
         POSTGRES_PASSWORD: heliora_pg_pass
      ports:
         - "5432:5432"
      volumes:
         - pg_data:/var/lib/postgresql/data

   redis:
      image: redis:7
      container_name: heliora-redis
      restart: unless-stopped
      ports:
         - "6379:6379"
      volumes:
         - redis_data:/data

   neo4j:
      image: neo4j:5
      container_name: heliora-neo4j
      restart: unless-stopped
      environment:
         NEO4J_AUTH: neo4j/heliora_neo4j_pass
      ports:
         - "7474:7474"
         - "7687:7687"
      volumes:
         - neo4j_data:/data
         - neo4j_logs:/logs

   rabbitmq:
      image: rabbitmq:3-management
      container_name: heliora-rabbitmq
      restart: unless-stopped
      environment:
         RABBITMQ_DEFAULT_USER: heliora
         RABBITMQ_DEFAULT_PASS: heliora_rmq_pass
      ports:
         - "5672:5672"
         - "15672:15672"
      volumes:
         - rabbitmq_data:/var/lib/rabbitmq

   minio:
      image: minio/minio:latest
      container_name: heliora-minio
      restart: unless-stopped
      command: server /data --console-address ":9001"
      environment:
         MINIO_ROOT_USER: heliora
         MINIO_ROOT_PASSWORD: heliora_minio_pass
      ports:
         - "9000:9000"
         - "9001:9001"
      volumes:
         - minio_data:/data

volumes:
   pg_data:
   redis_data:
   neo4j_data:
   neo4j_logs:
   rabbitmq_data:
   minio_data:
YAML

docker compose up -d
docker compose ps
```

成功信号:
1. `docker compose ps` 显示 5 个服务状态为 running。
2. 不建议所有服务复用同一个密码，至少按服务拆分数据库、消息队列、对象存储密码。

## 9.5 校验服务可用性

在 Ubuntu 虚拟机内执行:

```bash
docker exec -it heliora-postgres pg_isready -U heliora -d heliora
docker exec -it heliora-redis redis-cli ping
curl -I http://127.0.0.1:7474
curl -I http://127.0.0.1:15672
curl -I http://127.0.0.1:9001
```

成功信号:
1. PostgreSQL 返回 accepting connections。
2. Redis 返回 PONG。
3. Neo4j、RabbitMQ、MinIO 控制台返回 HTTP 200 或 302。

## 9.6 端口转发（VirtualBox NAT）建议口径

已配置:
1. 2026 -> 22（SSH）
2. 8081 -> 8000（API）
3. 3003 -> 7474（Neo4j HTTP）
4. 3036 -> 7687（Neo4j Bolt）

建议补齐:
1. 5433 -> 5432（PostgreSQL）
2. 6380 -> 6379（Redis）
3. 5673 -> 5672（RabbitMQ AMQP）
4. 15673 -> 15672（RabbitMQ 管理台）
5. 19000 -> 9000（MinIO API）
6. 19001 -> 9001（MinIO 控制台）
